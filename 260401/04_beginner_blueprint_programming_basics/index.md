# 초급 4편. 블루프린트 기초 프로그래밍

[이전: 초급 3편](../03_beginner_class_structure_and_blueprint_classes/) | [허브](../) | [다음: 부록 1](../05_appendix_official_docs_reference/)

## 이 편의 목표

이 편에서는 블루프린트를 노드형 장난감이 아니라, `BeginPlay`, `Tick`, 실행 핀, 데이터 핀, 변수, `Delta Seconds`를 갖춘 정식 프로그래밍 인터페이스로 읽기 시작한다.
핵심은 “노드 이름 암기”보다 “실행 시점과 값 흐름을 읽는 법”을 붙잡는 데 있다.

`BeginPlay / Tick 구분 -> 실행 핀 / 데이터 핀 이해 -> Delta Seconds 기반 이동 -> 변수로 일반화 -> Play로 즉시 검증`

## 블루프린트는 “쉬운 대체재”가 아니라 노드형 프로그래밍이다

블루프린트는 C++ 대신 대충 쓰는 도구가 아니라, 순서 제어와 데이터 전달을 그래프로 표현하는 프로그래밍 인터페이스다.

![블루프린트 이벤트 그래프 기본 화면](../assets/images/blueprint-event-graph-basics.jpg)

즉 블루프린트에서 하는 일도 결국 같다.

- 순서를 제어한다
- 값을 읽고 계산한다
- 월드 상태를 바꾼다
- 결과를 바로 확인한다

## `BeginPlay`와 `Tick`은 실행 시점이 다르다

입문 단계에서 가장 먼저 익혀야 하는 차이는 이것이다.

- `BeginPlay`
  게임 시작 시 한 번 들어온다
- `Tick`
  매 프레임 반복된다

![이벤트 그래프에서 `Event Tick`을 확인하는 장면](../assets/images/blueprint-event-tick.png)

이 구분은 현재 프로젝트 C++에서도 거의 그대로 대응된다.
`virtual void BeginPlay() override;`와 `virtual void Tick(float DeltaTime) override;`가 같은 생명주기 역할을 맡는다.

## 실행 핀과 데이터 핀을 구분하면 그래프가 읽히기 시작한다

블루프린트가 처음 어려운 이유는 노드 수보다 “무엇이 순서이고 무엇이 값인가”가 한 화면에 같이 보이기 때문이다.

- 실행 핀
  어떤 순서로 노드가 도는지 결정
- 데이터 핀
  노드 사이에 어떤 값이 오가는지 전달

그래프를 읽을 때는 먼저 실행 선을 따라가고, 그다음 어떤 값이 어디로 들어가는지 보는 편이 훨씬 안정적이다.

## `AddActorWorldOffset`과 `Delta Seconds`는 시간 기준 이동을 만든다

첫 실습의 진짜 핵심은 이동 노드 이름보다 “속도 x 시간 = 이번 프레임 이동량”이라는 사고방식이다.
매 프레임 고정 숫자를 더하는 대신 `Delta Seconds`를 곱해야 프레임레이트와 무관한 초당 이동량이 만들어진다.

![블루프린트에서 Add Actor World Offset을 사용하는 장면](../assets/images/blueprint-add-actor-world-offset.jpg)

이 사고는 이후 이동, 회전, 보간, 타이머 처리 전부의 바탕이 된다.

## 변수는 상수를 치우고 클래스를 재사용 가능하게 만든다

강의 중반부터는 하드코딩된 숫자를 `MoveSpeed` 같은 변수로 바꾸는 흐름이 나온다.
이 변화는 보기 좋게 만드는 수준이 아니라, 그래프를 재사용 가능한 클래스 구조로 바꾸는 첫 단계다.

![`MoveSpeed` 변수와 `인스턴스 편집 가능` 옵션을 함께 확인하는 장면](../assets/images/blueprint-instance-editable-variable.png)

변수로 바꾸면 아래 장점이 생긴다.

- 값의 의미에 이름을 붙일 수 있다
- 인스턴스마다 다른 값을 줄 수 있다
- 디테일 패널에서 바로 조절할 수 있다
- 스폰 시 노출, 읽기 전용 같은 메타 설정도 가능하다

즉 블루프린트에서 변수는 단순 저장칸이 아니라, 에디터와 로직을 연결하는 다리다.

## `Play In Editor`는 시연이 아니라 디버깅 루프의 시작이다

언리얼 작업은 “만들고 끝”보다 “조금 만들고 바로 실행해서 확인”하는 짧은 루프가 훨씬 중요하다.

![Play In Editor로 블루프린트 결과를 확인하는 장면](../assets/images/play-in-editor-bptestactor.jpg)

이 흐름을 초반부터 익혀 두면 뒤의 플레이어, 발사체, 충돌, 애니메이션 블루프린트 강의도 훨씬 수월해진다.

## 현재 프로젝트와 함께 읽는 포인트

첫날 강의는 블루프린트 중심이지만, 지금 프로젝트 `UE20252`도 결국 같은 생명주기를 쓴다.
`APlayerCharacter`, `AMonsterBase`, `AShinbi` 모두 `BeginPlay`, `Tick`, 입력 함수 바인딩을 통해 블루프린트와 거의 같은 이름의 실행 시점을 그대로 활용한다.

즉 첫날 블루프린트 감각을 제대로 잡아 두면, 나중에 C++ 소스를 읽을 때도 갑자기 완전히 다른 언어처럼 느껴지지 않는다.

## 이 편의 핵심 정리

1. 블루프린트는 노드형 프로그래밍이지 “쉬운 장난감”이 아니다.
2. `BeginPlay`와 `Tick` 차이를 이해해야 실행 시점이 잡힌다.
3. 실행 핀과 데이터 핀을 구분하면 그래프가 읽히기 시작한다.
4. `Delta Seconds`는 프레임 기준이 아니라 시간 기준 사고를 만들기 위한 핵심 값이다.
5. 변수와 인스턴스 편집 가능 설정은 블루프린트를 재사용 가능한 클래스로 바꾼다.
