# 초급 1편. 프로젝트 생성과 시작 규칙

[허브](../) | [다음: 초급 2편](../02_beginner_unreal_editor_basics/)

## 이 편의 목표

이 편에서는 첫 프로젝트를 만드는 순간에 왜 `템플릿`, `Blueprint / C++`, `경로`, `이름` 선택이 중요한지 정리한다.
핵심은 프로젝트 생성이 “에디터를 여는 클릭 절차”가 아니라, 이후 소스 구조와 빌드 안정성을 결정하는 시작점이라는 점이다.

## 전체 흐름 한 줄

`프로젝트 브라우저 -> 템플릿 선택 -> Blueprint / C++ 선택 -> 경로와 이름 규칙 -> 에디터/빌드 단위 이해`

## 프로젝트 브라우저는 이후 구조의 출발점이다

강의는 프로젝트 브라우저를 여는 데서 시작하지만, 실제로는 여기서부터 작업 습관이 정해진다.
어떤 템플릿을 고를지, `Blueprint`로 시작할지 `C++`로 시작할지, 프로젝트를 어느 폴더에 둘지 같은 선택이 이후 빌드, 폴더 구조, 협업 안정성과 직접 연결되기 때문이다.

![프로젝트 브라우저와 템플릿 선택 화면](../assets/images/project-browser-template.jpg)

즉 첫 강의의 핵심은 “프로젝트를 어떻게 열까”보다 “프로젝트를 어떻게 잘 시작할까”에 가깝다.

## 템플릿은 최종 구조가 아니라 학습용 발판이다

강의에서 여러 템플릿을 훑어보라고 하는 이유도 여기에 있다.
언리얼 템플릿은 완성된 정답 프로젝트보다, 각 장르에서 어떤 기본 액터와 카메라, 입력 구성이 들어가는지 보여 주는 출발점이다.

- `Blank`
  거의 비어 있는 시작점
- `First Person`
  1인칭 카메라와 기본 이동 구조
- `Third Person`
  3인칭 캐릭터와 추적 카메라

즉 템플릿은 “이걸 그대로 쓴다”가 아니라 “어떤 기본 구조를 참고하며 출발할 것인가”를 정하는 선택지다.

## `Blueprint`와 `C++`는 양자택일이 아니라 두 층이다

입문자가 가장 많이 오해하는 지점이 바로 이것이다.
프로젝트 생성 단계에서 `Blueprint`와 `C++`를 고르지만, 실제 언리얼 프로젝트는 둘을 함께 쓴다.

- `Blueprint`
  시각적 스크립팅과 에디터 조정에 강하다
- `C++`
  구조 제어와 소스 기반 확장에 강하다

현재 `UE20252`도 이 구조를 그대로 따른다.
월드 기본 규칙은 C++ `ADefaultGameMode`가 잡고, 실제 레벨과 자산 편집은 블루프린트와 에디터 자산이 이어서 담당한다.

```cpp
DefaultPawnClass = AShinbi::StaticClass();
PlayerControllerClass = AMainPlayerController::StaticClass();
```

즉 언리얼에서 `Blueprint`와 `C++`는 경쟁 관계가 아니라 역할 분담 관계다.

## 경로와 이름 규칙은 빌드 안정성과 연결된다

강의가 한글 경로, 너무 긴 경로, 임시 위치 사용을 피하라고 강조하는 이유도 단순 취향 문제가 아니다.
경로와 이름은 이후 Visual Studio, 빌드 툴, 플러그인, 백업, 협업 구조에 모두 영향을 준다.

프로젝트 경로가 불안정하면 나중에 다음 문제가 쉽게 생긴다.

- 빌드 도구 경로 인식 문제
- 외부 라이브러리/플러그인 경로 문제
- 사용자별 다른 위치로 인한 협업 혼선
- 소스와 에셋 백업 구조의 불안정

즉 첫날 폴더 규칙은 뒤의 모든 날짜를 위한 안전장치다.

## 프로젝트 생성 직후 에디터가 뜬다는 것은 에셋 작업과 빌드 단위가 함께 생겼다는 뜻이다

언리얼 프로젝트는 단순히 에셋 폴더 하나를 만드는 작업이 아니다.
`.uproject`와 에디터 작업 단위, 그리고 필요할 때는 IDE/빌드 구조까지 함께 묶인 개발 단위를 만드는 과정에 가깝다.

![프로젝트 생성 후 처음 열리는 에디터 화면](../assets/images/new-project-editor.jpg)

그래서 첫날부터 경로와 이름을 바르게 잡아 두는 것이 중요하다.
뒤에서 C++ 클래스를 추가하거나 프로젝트 설정을 늘릴수록 이 차이가 더 크게 드러난다.

## 현재 프로젝트 기준으로 다시 읽기

지금 프로젝트 `UE20252`는 GAS와 몬스터 AI까지 커졌지만, 출발선의 논리는 그대로다.
처음에 프로젝트를 어떤 규칙으로 만들고, 월드 기본 폰과 컨트롤러를 어떻게 연결할지 정하는 감각은 지금도 `ADefaultGameMode`에서 그대로 반복된다.

즉 `260401`은 입문 강의지만, 나중에 `AShinbi`, `AMainPlayerController`, `MonsterBase` 같은 소스를 읽을 때도 계속 다시 돌아오게 되는 출발점이다.

## 이 편의 핵심 정리

1. 프로젝트 생성은 단순 클릭 절차가 아니라 개발 환경의 뼈대를 정하는 일이다.
2. 템플릿은 학습용 발판이지 최종 구조가 아니다.
3. `Blueprint`와 `C++`는 둘 중 하나를 버리는 선택이 아니라 함께 쓰는 두 층이다.
4. 경로와 이름 규칙은 나중의 빌드와 협업 안정성을 위해 초반부터 중요하다.
